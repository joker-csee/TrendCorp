## P1-2 增加业务规则测试（Business Rule Test）

> 当前测试覆盖了算法（评分计算、排序），但对"最多保留 2 只中军""仓位不超过 70%""风险触发后禁止加仓"这类业务规则缺乏专项测试。

**决策：采纳。**

**理由**：

评审人指出了一个关键区别——"算法正确 ≠ 需求满足"。例如：
- Scanner 的加权评分算对了（算法正确），但如果返回了 5 只中军而不是 ≤2 只（违反业务规则），系统仍然不可用。
- RiskController 的熔断逻辑算对了，但如果熔断后 PositionManager 仍然允许开新仓（业务规则漏了），风控形同虚设。

这些业务规则是 ARCHITECTURE.md 中明确定义的"硬编码约束"，必须有专项测试。

**修改计划**：

在 PROJECT_PLAN.md 中新增"业务规则测试"章节，包含以下测试用例：

```text
BRT-01: 中军数量上限
  规则：每个板块最多筛选 2 只中军
  输入：板块内有 20 只评分 ≥ 0.65 的标的
  预期：CoreScreener.screen() 只返回 2 只，按 total_score 降序取前 2

BRT-02: 总仓位上限
  规则：任何时候总仓位 ≤ 70%
  输入：当前仓位 65%，请求建仓 15%
  预期：PositionManager 将建仓比例裁剪到 5%（65% + 5% = 70%）

BRT-03: 单票仓位上限
  规则：单票仓位 ≤ 30%
  输入：当前持仓某票 25%，请求加仓至 35%
  预期：PositionManager 将目标裁剪到 30%，实际加仓 5%

BRT-04: 单板块仓位上限
  规则：单板块所有持仓合计 ≤ 35%
  输入：AI 算力板块当前持仓合计 30%，请求在同一板块新增 15% 仓位
  预期：PositionManager 将新增仓位裁剪到 5%

BRT-05: 熔断后禁止开仓
  规则：熔断状态 ∈ {DAILY_BAN, WEEKLY_BAN, MONTHLY_BAN} 时，禁止任何 BUY 指令
  输入：当前风控状态 = DAILY_BAN，请求 A 级买点建仓
  预期：PositionManager 返回 None（不生成任何交易指令）

BRT-06: 止损优先级高于一切
  规则：当 L1 熔断（个股 -8%）触发时，即使 MA10/MA21 未破位也必须清仓
  输入：浮亏 -8.5%，MA10/MA21 均未破位
  预期：RiskController 返回 FuseLevel.STOCK_STOP，OrderExecutor 生成市价清仓指令

BRT-07: 评分阈值淘汰
  规则：中军评分 < 0.65 的标的不得入选
  输入：10 只标的，其中 7 只评分 < 0.65，3 只 ≥ 0.65
  预期：CoreScreener.screen() 最多返回 3 只（且全部 ≥ 0.65）

BRT-08: 每日最低现金保留
  规则：任何操作后，现金不得低于总净值的 30%
  输入：可用现金 = 32%，请求建仓总仓位 10%
  预期：PositionManager 将建仓比例裁剪到 2%（32% - 2% = 30% 最低现金）
```

这些测试不同于单元测试——它们是**跨模块的规则验证**，涉及多个 Engine 的协作。

## P1-3 增加配置驱动测试（Configuration Test）

> 核心行为由 config.py 中的权重和阈值控制，但当前没有验证"修改配置后系统行为是否按预期变化"。

**决策：采纳。**

**理由**：

评审人的例子很直观——如果把趋势权重设为 100%、其他三个维度权重设为 0%，总分应该完全等于趋势评分。这是对"配置真正驱动了行为"的最强验证。当前测试用的是默认配置，从未验证过"改了配置，系统真的会变"。

**修改计划**：

```text
CFG-01: 极端权重 — 单维度 100%
  配置：trend_weight=1.0, rel_strength=0, fund=0, echelon=0
  输入：板块 A, trend=0.8, rel=0.5, fund=0.6, echelon=0.7
  预期：total_score = 0.8（完全等于趋势评分 × 1.0）

CFG-02: 阈值调整 — 候选门槛
  配置：candidate_threshold=0.5（默认 0.6）
  输入：一个 total_score=0.55 的板块（默认配置下被过滤）
  预期：0.55 ≥ 0.5，该板块出现在候选列表中

CFG-03: 阈值调整 — 中军门槛
  配置：core_score_threshold=0.80（默认 0.65）
  输入：一批 total_score 在 0.65-0.79 的标的
  预期：全部被过滤，无中军入选

CFG-04: 仓位上限调整
  配置：total_cap=0.50（默认 0.70）
  输入：当前仓位 45%，请求建仓 15%
  预期：裁剪到 5%（45% + 5% = 50%），拒绝超过 50%

CFG-05: 止损线调整
  配置：stock_loss_limit=-0.05（默认 -0.08）
  输入：浮亏 -6%
  预期：L1 熔断触发（-6% < -5%）

CFG-06: 权重之和不为 1 时的处理
  配置：trend=0.3, rel=0.3, fund=0.3, echelon=0.3（合计 1.2）
  预期：系统启动时 WARNING 日志，或自动归一化，不静默忽略

CFG-07: 无效配置值
  配置：total_cap=1.5（> 100%）
  预期：系统启动时 ERROR 日志，拒绝加载，使用默认值
```

## P2-1 Repository 提前到 M1

> 当前 Repository 安排在 M2，但 M1 已有数据库和 Provider。建议提前到 M1，避免开发早期形成"直接操作 SQLite"的习惯。

**决策：采纳，调整为 M1 实现核心 Repository。**

**理由**：

评审人预判的"直接操作 SQLite 再重构"是真实风险。M1 的 Provider 拉取数据后需要落库，如果没有 Repository 层，开发人员（包括 AI Agent）会直接在 Provider 或测试脚本中写 `sqlite3.execute(...)`。一旦形成模式，M2 再抽 Repository 会变成"找所有散落的 SQL 语句并替换"——增加不必要的返工。

**修改计划**：

M1 任务表中增加：
- `T1.10a` — `SectorRepository`（1.5h，紧接 T1.4 建表）
- `T1.10b` — `StockRepository`（1h）
- `T1.10c` — `NavRepository`（0.5h）

其余 Repository（trade/position/risk/core_score）在 M2 随对应 Engine 实现时补充。

M1 总预估从 20.5h → 23.5h（+3h）。

## P2-3 增加 Workflow 调用验证测试

> 当前 Workflow 测试主要是端到端（验证结果正确），缺少"中间步骤的调用顺序和完整性"的验证。

**决策：采纳，M3 增加调用链验证测试。**

**理由**：

端到端测试的问题在于——如果 Workflow 跳过了"保存快照"步骤但最终输出仍正确（因为数据直接在内存中传递），端到端测试不会捕获这个遗漏。但"保存快照"是否执行，决定了复盘时能否回溯历史——这是功能正确性的一部分，只是延迟暴露。

**修改计划**：

```text
TC-3.10: Workflow 调用完整性 — Weekly
  前置：使用 Mock 对象替代所有 Engine/Provider/Repository
  步骤：调用 WeeklyWorkflow.run()

  预期：
  1. MarketProvider.fetch_all_sectors() 被调用 1 次
  2. MarketScanner.scan_all() 被调用 1 次，参数为当日日期
  3. ThemeSelector.confirm() 被调用 1 次，参数为 Scanner 的输出
  4. CoreScreener.screen() 被调用 N 次（N = 确认主线数量）
  5. SectorRepository.save_snapshot() 被调用 ≥ 1 次
  6. CoreScoreRepository.save_snapshot() 被调用 ≥ 1 次
  7. 以上调用按 1→2→3→4→5 顺序执行
  8. 中间任何一步失败，后续步骤不执行（异常传播正确）

TC-3.11: Workflow 调用完整性 — Daily
  同上模式，验证 DailyWorkflow 的 5 个步骤全部被调用且顺序正确

TC-3.12: Workflow 步骤失败不静默
  前置：Mock 对象，令 Scanner.scan_all() 抛出异常
  步骤：调用 WeeklyWorkflow.run()

  预期：
  - ThemeSelector.confirm() 从未被调用
  - CoreScreener.screen() 从未被调用
  - 异常被重新抛出，error.log 有记录
```

# 结语

评审人从"需求质量"角度补齐了原计划中最薄弱的一环——**测试覆盖了算法，但没有显式覆盖业务规则**。BRT-01~08 那 8 条测试，每一条都直接对应 ARCHITECTURE.md 中定义的一个硬约束，这些约束如果被违反，比"权重算错了 0.01"更致命。

另外，需求追踪矩阵的价值不在 V1——在 V3。当你半年后想"把单票上限从 30% 改到 25%"，RTM 告诉你这一行配置影响 REQ-15/BRT-03/CFG-04 三条需求和两个测试——你不需要翻遍 1000 行代码来找影响范围。

这是这次评审最有价值的两个建议。
