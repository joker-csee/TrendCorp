## P1-1 建立需求追踪矩阵（RTM）

### 问题

目前：

- 有需求
- 有架构
- 有开发任务
- 有测试

但是四者之间没有明确关联。

例如：

看到一个测试：

```text
TC-2.11
```

无法立即知道：

对应哪个需求。

也无法知道：

对应哪个模块。

------

### 建议

增加一张：

Requirement Traceability Matrix

例如：

| Requirement | Architecture | Task | Test       |
| ----------- | ------------ | ---- | ---------- |
| 板块扫描    | scanner      | T2.x | TC2.1~2.11 |
| 主线确认    | selector     | T2.x | TC2.18     |
| 中军筛选    | screener     | T2.x | TC2.15~17  |
| Workflow    | workflow     | T3.x | TC3.x      |

以后：

任何需求修改。

立即知道：

哪些地方需要修改。

------

### 原因

这是整个项目后续维护收益最大的改进。

尤其适合 AI Agent 开发。

------

## P1-2 增加业务规则测试（Business Rule Test）

### 当前情况

目前：

测试更多关注：

算法。

例如：

```text
MA计算

评分计算

排序
```

但是：

真正重要的是：

业务规则。

例如：

需求：

```text
最多保留两个中军
```

应该有：

```text
输入：

10只股票

预期：

返回2只
```

类似：

```text
仓位不能超过70%

风险触发后禁止加仓

评分低于阈值直接淘汰
```

都属于：

业务规则。

------

### 建议

增加：

Business Rule Test

覆盖：

所有需求中的业务限制。

------

### 原因

业务规则才是真正的软件需求。

算法只是实现方式。

------

## P1-3 增加配置驱动测试

目前：

很多核心逻辑：

来自：

config.py

例如：

- 权重
- 阈值
- 仓位比例

但是：

目前没有验证：

修改配置后。

系统是否正常变化。

例如：

```text
趋势权重

100%

其它

0%
```

结果：

总分。

是否：

完全等于趋势评分。

------

建议：

增加：

Configuration Test。

验证：

配置修改。

不会导致系统异常。

------

### 原因

项目强调：

配置驱动。

测试也应该覆盖配置。

# 六、评审总结

本次评审认为，**PROJECT_PLAN 与 SYSTEM_ARCHITECTURE 保持了较高的一致性**，开发任务能够完整映射架构设计，测试体系也已覆盖主要模块和流程，整体已经具备作为 AI Agent 开发依据的质量。

后续优化重点不应放在增加新的架构层或技术栈，而应放在**提升需求的可追踪性和测试对需求的覆盖能力**。其中，**建立需求追踪矩阵（RTM）**、**补充业务规则测试**和**增加配置驱动测试**是收益最高、最符合项目长期演进目标的三项改进。这些改进能够显著降低未来需求变更、功能扩展和回归验证的成本，同时保持系统架构的简洁性和可维护性。
